## Bug Fix: Calendar Events Off-by-One Day

**Problem:** All calendar events were displaying one day ahead. For example, a Booking.com reservation ending April 14 was shown ending April 15 on the dashboard.

**Root Cause:** The `localDateWithTime()` function in both `worker/src/ics.js` and `src/calendarService.js` used JavaScript's local time methods (`.getFullYear()`, `.getMonth()`, `.getDate()`) on UTC date objects. When the server/worker timezone is ahead of UTC (e.g., Europe/Sofia = UTC+3), midnight UTC dates like `2026-04-14T21:00:00.000Z` were interpreted as April 15 in local time instead of April 14 in UTC.

**Fix:** Changed to UTC date methods (`.getUTCFullYear()`, `.getUTCMonth()`, `.getUTCDate()`) in both files:
- `worker/src/ics.js` (Cloudflare Workers production)
- `src/calendarService.js` (Express local dev)

**Files Changed:**
- `worker/src/ics.js` — line 148-152: `localDateWithTime()` uses UTC methods
- `src/calendarService.js` — line 50-56: `localDateWithTime()` uses UTC methods

**Validation:**
- Local server restarted and `/api/events` confirmed: `rawEnd: "2026-04-14T21:00:00.000Z"` → `end: "2026-04-14T11:00"` (was previously `2026-04-15T11:00`)
- Property correctly shows free after April 14 at 11:00am, matching Airbnb source calendar

## Fix: Cloudflare Worker 500 Errors on /api/version and SPA Routes

**Problem:** The live Cloudflare Workers deployment returned 500 errors on `GET /api/version` and SPA fallback routes like `/en/reservations`.

**Root Cause:** Two issues in `worker/src/index.js`:
1. `/api/version` used `new Request(new URL(...))` to fetch from `env.ASSETS` — the Request constructor with a URL object may not work correctly with the Workers ASSETS binding.
2. SPA fallback passed the original `request` as the second argument to `new Request()`, copying method/headers/body from the original request which could cause incompatibilities with static asset fetching.

**Fix:** Changed both to pass `url.toString()` (a plain string) to `env.ASSETS.fetch()` instead of constructing a new Request object:
- `/api/version`: `env.ASSETS.fetch(appUrl.toString())`
- SPA fallback: `env.ASSETS.fetch(indexUrl.toString())`

**Files Changed:**
- `worker/src/index.js` — simplified ASSETS.fetch calls

**Fix (updated):** The actual root cause was that `env.ASSETS` was `undefined` because `wrangler.jsonc` had `assets.directory` but no `assets.binding`. Added `"binding": "ASSETS"` to make the ASSETS fetcher available in worker code.

**Files Changed:**
- `wrangler.jsonc` — added `"binding": "ASSETS"` to assets config
- `worker/src/index.js` — cleaned up debug error handling

**Validation:**
- `npm run build` succeeds
- Deployed via git push (auto-deploys from main branch)
