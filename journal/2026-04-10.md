## Bug Fix: Calendar Events Off-by-One Day

**Problem:** All calendar events were displaying one day ahead. For example, a Booking.com reservation ending April 14 was shown ending April 15 on the dashboard.

**Root Cause:** The `localDateWithTime()` function in both `worker/src/ics.js` and `src/calendarService.js` used JavaScript's local time methods (`.getFullYear()`, `.getMonth()`, `.getDate()`) on UTC date objects. When the server/worker timezone is ahead of UTC (e.g., Europe/Sofia = UTC+3), midnight UTC dates like `2026-04-14T21:00:00.000Z` were interpreted as April 15 in local time instead of April 14 in UTC.

**Fix:** Changed to UTC date methods (`.getUTCFullYear()`, `.getUTCMonth()`, `.getUTCDate()`) in both files:
- `worker/src/ics.js` (Cloudflare Workers production)
- `src/calendarService.js` (Express local dev)

**Files Changed:**
- `worker/src/ics.js` ‚Äî line 148-152: `localDateWithTime()` uses UTC methods
- `src/calendarService.js` ‚Äî line 50-56: `localDateWithTime()` uses UTC methods

**Validation:**
- Local server restarted and `/api/events` confirmed: `rawEnd: "2026-04-14T21:00:00.000Z"` ‚Üí `end: "2026-04-14T11:00"` (was previously `2026-04-15T11:00`)
- Property correctly shows free after April 14 at 11:00am, matching Airbnb source calendar

## Performance: Timezone Selector Moved Server-Side

**Problem:** Settings page loaded slowly because `initTimezone()` computed UTC offsets for 418 IANA timezones client-side on every page visit, calling `Intl.DateTimeFormat` for each one.

**Fix:** Pre-compute the timezone list server-side and serve via a new API endpoint. Frontend fetches the cached result instead of computing it.

**API Endpoint:** `GET /api/timezones`
- Returns JSON array: `[{ tz, label, off }, ...]` (418 entries, sorted by UTC offset)
- Server caches result in memory (computed once)
- `Cache-Control: public, max-age=86400` (24h browser cache)

**Files Changed:**
- `src/server.js` ‚Äî added `GET /api/timezones` with in-memory cache (`_tzCache`)
- `worker/src/index.js` ‚Äî added matching `GET /api/timezones` for Cloudflare Workers
- `src/public/app.js` ‚Äî `initTimezone()` now async, fetches from `/api/timezones` (or `data/timezones.json` in static mode). Removed `COMMON_TIMEZONES` constant and client-side `Intl.supportedValuesOf` computation
- `scripts/export-static.js` ‚Äî generates `docs/data/timezones.json` during static build

**Validation:**
- `/api/timezones` returns 418 timezones with correct offsets (e.g. `Europe/Sofia (GMT+3)`)
- `npm run build` succeeds, `docs/data/timezones.json` generated (31KB)
- Settings page timezone dropdown renders correctly from server data
